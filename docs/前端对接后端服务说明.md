# 前端对接后端服务 — 修改入口说明

本文说明：**从当前「Mock 接口（内存）」切换到真实 HTTP 后端**时，应改哪些文件；日常业务页面应如何通过统一客户端访问数据。

---

## 1. 一键切换 Mock / HTTP

| 配置项                                   | 位置                  | 作用                                                                                                  |
| ---------------------------------------- | --------------------- | ----------------------------------------------------------------------------------------------------- |
| `VITE_SKILL_MARKET_TRANSPORT`            | `.env` / 部署环境变量 | `mock`：内存 Mock；`https` / `http`：走真实网络请求                                                   |
| `VITE_SKILL_MARKET_API_BASE`             | 同上                  | 后端根地址，例如 `https://api.example.com`；留空表示与页面**同源**（由网关反代 `/api`）               |
| `VITE_SKILL_MARKET_MOCK_ROLE`            | 同上                  | **仅 `mock`**：模拟当前用户市场角色 `USER` / `ORG_ADMIN` / `SUPER_ADMIN`；不填则 **`ORG_ADMIN`**      |
| `VITE_SKILL_MARKET_MOCK_MANAGED_ORG_IDS` | 同上                  | **仅 `mock` 且角色为 `ORG_ADMIN`**：管辖组织 id（与种子组织表一致），逗号分隔；不填则**全部**种子组织 |

示例见项目根目录 **`.env.example`**。

当 **`VITE_SKILL_MARKET_TRANSPORT=https`**（或兼容旧配置 `http`）时，应用只使用 **`skillMarketHttpClient`**，**不会**实例化 Mock 客户端，因此 **`VITE_SKILL_MARKET_MOCK_ROLE` / `VITE_SKILL_MARKET_MOCK_MANAGED_ORG_IDS` 是否存在于 `.env` 均不影响**正式接口返回的角色与菜单；角色以服务端 **`/api/users/current/role`**（见 `endpoints.ts`）为准。

## 2. 必须了解的目录与职责（按修改频率排序）

### 2.1 接口路径（与《市场需求开发设计文档》对齐）

**文件：`src/services/skillMarket/endpoints.ts`**

- 所有 REST 路径常量集中在此（含 **`/api/users/current/role`**、**`/api/super-admins`** 等角色相关路径）。
- 若后端有**统一前缀**（如 `/gateway/v1`）或某条路径与设计文档不一致：**只改这里**（或在 `VITE_SKILL_MARKET_API_BASE` 里带前缀），避免在页面里散落硬编码 URL。

### 2.2 请求/响应类型（契约）

**文件：`src/services/skillMarket/apiTypes.ts`**

- 标准响应壳 `ApiEnvelope<T>` 及各接口 `data` 的 DTO。
- 后端字段变更时：先改类型与文档，再改 **HTTP 实现**里的解析/映射。

### 2.3 对外能力接口（页面与 Store 只依赖它）

**文件：`src/services/skillMarket/skillMarketClient.types.ts`**

- 类型名：**`SkillMarketClient`**。
- 新增/变更业务能力：在此增加或修改方法签名，并同时实现：
  - **`src/services/skillMarket/skillMarketMockClient.ts`**（Mock）
  - **`src/services/skillMarket/skillMarketHttpClient.ts`**（HTTP）

### 2.4 HTTP 实现（真实联调主改这里）

**文件：`src/services/skillMarket/skillMarketHttpClient.ts`**

- `fetch`、`FormData`、`URL` 查询参数、错误处理等。
- 后端鉴权头、错误码与 `ApiEnvelope` 不一致时：**主要改此文件**。
- 运营管理 UI 数据（设计文档 **§3.3.13**）：
  - **扶摇**：请求 **`GET /api/dashboard/overview?system=fuyao`**，响应 **`DashboardOverviewDto`**；**`fetchOpsDashboardUi('fuyao')`** 内用 **`dashboardOverviewToOpsBundle`** 转为 **`OpsDashboardBundle`**。UI 上**不展示**数据口径说明（含 T+1）、**不展示** Excel 导入（仅公司系统展示）；**不调** **`POST /api/dashboard/import-excel`**。
  - **公司系统**：**不单独请求看板接口**；只读打包的 **`src/mock/opsDashboardCompanyDefault.json`**。UI 展示 **T+1** 口径说明；**Excel 导入**仅页面预览并下载 JSON，由你手动替换 mock 文件后重新 **dev/build** 生效。

### 2.5 Mock 实现（本地演示、单测）

**文件：`src/services/skillMarket/skillMarketMockClient.ts`**

- 内存状态、与文档一致的返回形状。
- **`fetchCurrentUserRole`**：在 **`VITE_SKILL_MARKET_MOCK_ROLE`**（及可选 **`VITE_SKILL_MARKET_MOCK_MANAGED_ORG_IDS`**）控制下返回 `CurrentUserRoleDto`，便于本地验证菜单与按钮权限。
- **`fetchOrganizations`**：与 HTTP 约定一致——`USER` 返回空数组；`ORG_ADMIN` 仅返回 `managedOrgIds` 范围内组织；`SUPER_ADMIN` 返回全量。`postOrganization` / `putOrganization` 在 Mock 中按相同规则返回 403。
- 仅本地演示数据：**`src/services/skillMarket/mock/builtInSkills.ts`**（内置 Skill 列表）；运营管理 Mock：**公司** 与 HTTP 一致，均只 **`readOpsDashboardBundleFromJson('company')`**；**扶摇** 走 **`fetchDashboardOverview`** 再映射；**扶摇** Excel 导入不写内存、不覆盖看板。

### 2.6 工厂（选择 Mock / HTTP）

**文件：`src/services/skillMarket/createSkillMarketClient.ts`**

- 一般无需改；除非增加第三种传输方式（如 WebSocket）。

### 2.7 UI 模型与 DTO 映射

**文件：`src/services/skillMarket/mappers.ts`**、**`src/services/skillMarket/parseOpsDashboardApiResponse.ts`**

- 列表/详情 DTO ↔ 页面使用的 `Skill` 等。
- 运营管理 HTTP 响应解析（与本地 JSON 同结构）：**`parseOpsDashboardApiResponse.ts`**。

### 2.8 聚合导出

**文件：`src/services/skillMarket/index.ts`**

- 对外 `import { createSkillMarketClient, SKILL_MARKET_ENDPOINTS, … } from '@/services/skillMarket'` 的入口。

## 3. 页面与 Store 应如何访问接口

- **推荐**：在组件中使用 **`const { marketClient, skills, … } = useSkillMarketStore()`**，通过 **`marketClient.xxx()`** 调用设计文档中的接口（如 `fetchOpsDashboardUi`、`postDashboardImportExcel`、`fetchSyncApplications` 等）。
- **市场 Skill 列表/上传/下载**：已通过 Store 封装为 `uploadSkill` / `downloadSkill` / `skills`，内部仍走同一 `SkillMarketClient`。

**不要**在 `.vue` 里直接 `import` `src/mock/*.json` 作为业务数据源；默认看板数据已由 **`fetchOpsDashboardUi`** 经 Mock 客户端注入。

- **当前用户角色**：应用启动后 Store 会调用 **`refreshCurrentUserRole()`**（内部 **`GET /api/users/current/role`**）。使用 **`currentUserRole`** 与 **`src/services/skillMarket/roleUi.ts`** 中的辅助函数控制菜单/按钮展示（与文档 §3.3.10 菜单规则对齐；接口拒绝仍以服务端为准）。

## 3.1 角色模型与接口（设计文档 §1.5、§3.3.10–12、§3.4.3、§4.4 摘要）

### 三类角色

| 角色                     | 枚举 `role`   | 说明                                                                                                                          |
| ------------------------ | ------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| 超级管理员               | `SUPER_ADMIN` | 独立表 `t_skill_super_admin` 工号判定；可维护全部组织、**新增组织**、**超级管理员配置**                                       |
| 普通管理员（组织管理员） | `ORG_ADMIN`   | `organization.admins` 中英文字号逗号拆分精确匹配；仅可管 **`managedOrgIds`** 范围内组织，**不可新增组织**、**不可进超管配置** |
| 普通用户                 | `USER`        | 市场总览、我的发布、**运营管理（只读，无 Excel 导入）**；**不展示**组织管理 / 审核中心 / 质量评审 / 超管配置                  |

**优先级**：`SUPER_ADMIN` > `ORG_ADMIN` > `USER`（同一人同时在超管表与 `admins` 中时按超管）。

### 组织配置权限（`organizationScope`）

| 值            | 含义                                |
| ------------- | ----------------------------------- |
| `ALL`         | 超级管理员，组织接口可查改全部      |
| `MANAGED_ORG` | 普通管理员，仅 `managedOrgIds` 范围 |
| `NONE`        | 普通用户                            |

### 前端应对接的接口（已写入 `endpoints.ts` / `SkillMarketClient`）

| 接口                      | 方法       | 用途                                                                                                                                                      |
| ------------------------- | ---------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `/api/users/current/role` | GET        | 拉取 `CurrentUserRoleDto`，驱动菜单与「新增组织」等按钮                                                                                                   |
| `/api/super-admins`       | GET / POST | 超管列表 / 新增（**仅 SUPER_ADMIN**）                                                                                                                     |
| `/api/super-admins/{id}`  | PUT        | 启停、备注等（**仅 SUPER_ADMIN**）                                                                                                                        |
| `/api/organizations`      | GET / POST | **GET**：`USER` 宜返回 `[]`；`ORG_ADMIN` **仅返回** `managedOrgIds` 内组织（无权限的不出现在列表）；`SUPER_ADMIN` 返回可维护全量。POST **仅 SUPER_ADMIN** |
| `/api/organizations/{id}` | PUT        | `SUPER_ADMIN` 可改任意；`ORG_ADMIN` 仅能改管辖内组织，否则 403                                                                                            |

### 其它接口与角色（后端拦截，摘要）

- **审核同步申请**：目标组织 ∈ `managedOrgIds` 的 `ORG_ADMIN`，或 `SUPER_ADMIN`（§3.4.3）。
- **质量评审保存/归档**：普通用户不可调用（§3.3.14）。
- **运营管理 / 审核中心数据范围**：`SUPER_ADMIN` 全量；`ORG_ADMIN` 按组织范围（§4.4.3）。

### 前端辅助函数

**文件：`src/services/skillMarket/roleUi.ts`**

- `marketRoleShowsSuperAdminSettings`、`marketRoleCanCreateOrganization`、`marketRoleShowsOrgManagement` 等，与 §3.3.10 菜单规则一致，供 `.vue` 使用。

## 4. 联调真实后端时的最小检查清单

1. 设置 `VITE_SKILL_MARKET_TRANSPORT=https` 与正确的 `VITE_SKILL_MARKET_API_BASE`。
2. 对照设计文档核对 **`endpoints.ts`** 路径是否与网关一致。
3. 用浏览器网络面板看 **`skillMarketHttpClient.ts`** 发出的请求 URL、方法、Body 是否与后端一致。
4. 运营管理：**扶摇** 实现 **`GET /api/dashboard/overview?system=fuyao`**（§3.3.13）；**公司系统** 看板数据来自仓库内 **`opsDashboardCompanyDefault.json`** 打包进前端，无需 **`ops-ui`**；更新数据靠手动替换该文件。
5. 角色：实现 **`GET /api/users/current/role`** 及（若做超管配置页）**`/api/super-admins`**，字段与 **`apiTypes.ts`** 中 `CurrentUserRoleDto`、`SuperAdminDto` 一致。
6. 若其它接口响应与 **`apiTypes.ts`** 不一致：先更新类型，再改 **HTTP 客户端**解析或 **mappers**。
